# OpenFS 需求文档

## 一、项目概述

### 1.1 项目名称

OpenFS — AI-Native Distributed Storage System

### 1.2 项目定位

面向 AI 时代的高性能分布式存储系统，同时支持文件存储（POSIX）和对象存储（S3 兼容）语义，
以"Block 级调度 + 元数据驱动安全"为核心理念，在保证高可用的同时实现 90%+ 的存储空间利用率。

### 1.3 开发语言

C++ (C++17/20)

### 1.4 核心设计哲学

| 原则 | 说明 |
|---|---|
| Block 即一切 | 所有数据统一打散为 Block，Block 是最小调度单元 |
| 元数据驱动安全 | 不依赖数据冗余保安全，靠元数据校验 + 快速重建 |
| 顺序写入 + 延迟分散 | 节点内 Segment 顺序追加，节点间批次轮转 |
| 多级 Block 自适应 | 系统根据文件大小自动选择最优 Block 尺寸，用户零配置 |
| 节点越多越强 | 节点增加同时带来容量、性能、安全性三重提升 |

## 二、功能需求

### 2.1 存储语义支持

#### 2.1.1 文件存储（POSIX 语义）

- 支持标准 POSIX 文件操作：open / read / write / close / stat / mkdir / readdir / rename / unlink
- 通过 FUSE 挂载为本地文件系统，对应用透明
- 支持目录树层级结构
- 支持文件权限管理（owner / group / mode）

#### 2.1.2 对象存储（S3 兼容）

- 支持 S3 兼容 API：PutObject / GetObject / DeleteObject / ListObjects / HeadObject
- 支持 Bucket 管理：CreateBucket / DeleteBucket / ListBuckets
- 支持 Multipart Upload（大对象分段上传）
- 支持 Range Read（范围读取）

### 2.2 集群架构

#### 2.2.1 元数据集群（MetaCluster）

- 独立的元数据服务集群
- 最少 1 个节点可运行（单节点模式退化为单机存储）
- 3+ 节点实现高可用（Raft 共识协议）
- 可水平扩展，通过 Subtree Partitioning（子树分区）将不同命名空间/目录子树分配到不同 Raft Group
- 存储内容：
  - 文件/对象的 inode 信息（大小、权限、时间戳）
  - 目录树结构
  - Block 映射关系（FileID → BlockID 列表）
  - Block 位置信息（BlockID → NodeID + SegmentID + Offset）
  - Block 校验和（CRC32/SHA256）
  - 数据节点状态与拓扑信息

#### 2.2.2 数据集群（DataCluster）

- 独立的数据存储集群
- 最少 1 个节点可运行
- 无状态设计，每个 DataNode 只负责 Block 的存取
- 可随时加入新节点，自动注册到元数据集群
- 新节点加入后自动触发后台 Rebalance
- 节点数量几乎无上限，可自由水平扩展

### 2.3 数据组织

#### 2.3.1 多级 Block 自适应

系统定义 5 级 Block 尺寸，写入时根据文件大小自动选择：

| Block 等级 | 大小 | 适用场景 |
|---|---|---|
| Level 0 | 64KB | 小文件（缩略图、配置文件） |
| Level 1 | 512KB | 中小文件（普通图片、文档） |
| Level 2 | 4MB | 中等文件（高清图片、小视频） |
| Level 3 | 32MB | 大文件分片（数据集、镜像层） |
| Level 4 | 256MB | 超大文件分片（大模型、原始数据集） |

自动选择规则：

| 文件大小 | Block 策略 |
|---|---|
| ≤ 64KB | 整文件作为 L0 Block，进入 PackedBlock 打包 |
| ≤ 512KB | 整文件作为 L1 Block |
| ≤ 4MB | 整文件作为 L2 Block |
| ≤ 32MB | 整文件作为 L3 Block |
| > 32MB | 按 L3 或 L4 切分为多个 Block |
| 流式写入(大小未知) | 先用 L2 切分，seal 时可合并升级 |

#### 2.3.2 小文件打包（PackedBlock）

- 多个小文件（≤ 64KB）合并打包进一个物理 Block（4MB）
- 元数据记录每个小文件在 PackedBlock 内的偏移和大小
- 消除小文件的空间碎片，利用率接近 100%

#### 2.3.3 Segment 日志式存储

DataNode 本地使用 Segment 文件组织数据：

- Segment 固定大小 256MB
- Block 以追加方式顺序写入 Segment（纯顺序写）
- Segment 写满后封存（seal），变为只读不可变
- 不同级别的 Block 可混合存入同一 Segment
- 每个 Segment 维护内部索引（BlockID → Offset 映射）

### 2.4 写入策略：追加批次分条（Append-Batched-Striping）

- Client 缓冲一批 Block（如 16 个，约 64MB）
- 元数据分配写入批次，选定目标 DataNode
- 整批 Block 以追加方式顺序写入目标节点的活跃 Segment
- 下一批次轮转到另一个节点
- 节点间批次级别轮转保证负载均衡
- 节点内 Segment 顺序追加保证磁盘顺序性

### 2.5 数据安全性（元数据驱动）

#### 2.5.1 数据校验链

- 写入时：Client 计算 CRC32 → 随 Block 发送 → DataNode 校验并存储
- 元数据记录：每个 BlockID 对应的 CRC32 + 写入时间戳 + NodeID
- 读取时：DataNode 读出 → 校验 CRC32 → 不匹配则报告元数据触发修复
- 后台定期巡检：逐 Block 校验，发现静默损坏立即修复

#### 2.5.2 数据保护策略（按节点规模自适应）

| 节点数 | 策略 | 存储利用率 |
|---|---|---|
| 1 | 本地双副本 | ~50% |
| 3 | 关键 Block 双副本 | 70-80% |
| 10+ | 热 Block 选择性双副本 | 85-90% |
| 50+ | 全单副本 + 元数据保障 | 92-95% |
| 100+ | 纯单副本 | 95%+ |

#### 2.5.3 故障检测与自愈

- DataNode 定期向元数据上报心跳 + Block 清单 + 磁盘使用率
- 心跳超时标记节点下线
- 元数据立即定位受影响的 Block
- 从其他节点的副本/存活数据快速重建到新节点
- 全程自动，无需人工干预

#### 2.5.4 快速重建（核心优势）

重建时间公式：
```
重建时间 ≈ 单节点数据量 / (剩余节点数 × 单节点带宽)
```
节点越多 → 重建越快 → 数据越安全 → 越不需要副本

### 2.6 热度感知调度

- 元数据跟踪 Block 访问频次
- 热 Block 自动增加副本到就近节点（读缓存加速）
- 冷 Block 保持单副本，极致省空间
- 后台可对冷 PackedBlock 执行 Compact 重整

### 2.7 多级缓存体系

#### 2.7.1 三级缓存架构

| 缓存层 | 位置 | 介质 | 延迟 | 容量级别 |
|---|---|---|---|---|
| L1 | Client 侧 | DRAM | < 1μs | GB 级 |
| L2 | DataNode | DRAM | ~10μs | 10GB 级 |
| L3 | DataNode | NVMe | ~100μs | TB 级 |
| Cold Store | DataNode | HDD | ~5ms | 10TB+ |

#### 2.7.2 缓存一致性

- **元数据驱动**：缓存有效性由元数据的版本号（version）裁决
- **Version + Lease 机制**：Client 缓存携带版本号和租约，过期自动校验
- **Block 不可变特性**：Block 一旦写入不可修改，DataNode 侧缓存天然一致
- **写入通知失效**：文件修改后 MetaNode 主动推送 Invalidate 到其他 Client

#### 2.7.3 写入缓存模式

- **Write-Through（默认）**：数据直接写入持久化 Segment，强一致，适用于关键数据
- **Write-Back（可选）**：数据先写入 NVMe 缓存，后台异步刷入 HDD，高性能，适用于临时数据
- 写入模式由元数据根据文件路径/标签自动选择，用户也可显式指定

#### 2.7.4 硬件感知与自适应

- DataNode 启动时自动探测存储设备类型（NVMe / SATA SSD / HDD）和性能基线
- 系统根据硬件配置自动分配角色（缓存层 / 存储层）
- 支持多种硬件架构：NVMe 缓存 + HDD 存储、全闪存、混合集群

#### 2.7.5 缓存预热

- **主动预热**：CLI 命令触发指定目录的缓存预热（训练前准备）
- **智能预热**：系统检测顺序访问模式时自动预读到 NVMe 缓存
- **跨节点预热**：任务迁移时自动在目标节点预热所需数据

### 2.8 面向 AI 场景优化

| 场景 | 优化策略 |
|---|---|
| 训练数据批量加载 | Pipeline 多 Block 并行预读 + Shuffle Read + 缓存预热 |
| 模型 Checkpoint | Append-Only 大 Block 写入 + Write-Through 持久化 |
| 推理小文件随机读 | Client 侧元数据缓存 + L2/L3 热数据缓存 |
| 大文件顺序读 | 多节点并行拉取 + L3 NVMe 预读 |
| 训练 epoch 迭代 | 首 epoch 数据自动提升到 NVMe 缓存，后续 epoch 全缓存命中 |

## 四、接口需求

### 4.1 Client SDK（libclient）

- C++ 原生 SDK
- POSIX 文件接口封装
- S3 兼容接口封装
- 支持连接池、重试、超时配置
- 支持客户端侧元数据缓存

### 4.2 FUSE 接口

- 基于 libfuse3 实现
- 挂载为本地文件系统
- 支持多线程并发

### 4.3 S3 Gateway

- HTTP 服务，兼容 S3 协议
- 支持常用 S3 操作
- 可独立部署，无状态，可水平扩展

### 4.4 管理接口

- gRPC 管理 API
- 集群状态查询
- 节点管理（添加/移除/状态）
- 存储空间统计
- 命令行管理工具（openfs-cli）

## 五、约束与限制

### 5.1 技术约束

- 开发语言：C++ (C++17/20 标准)
- 构建系统：CMake
- 操作系统：Linux（主要目标平台）
- 网络协议：gRPC / brpc（节点间通信）
- 共识协议：Raft（元数据一致性）
- 本地存储引擎：RocksDB（元数据持久化）
- 异步 IO：io_uring（数据节点 IO）

### 5.2 设计约束

- 元数据和数据完全分离，独立扩展
- Block 为最小调度单元，不绑定文件、不绑定节点
- 数据路径直连（Client 直接读写 DataNode，不经过 MetaNode 转发）
- 写入路径保证磁盘顺序性（Segment 追加写入）
